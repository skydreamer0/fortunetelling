# 命理綜合分析平台 — 架構計畫書（供外部審計）

> **給審計者**：本文件自成一體，你不需要存取原始碼即可審閱。目的是請你審計
> ①架構是否合理且可多人重複使用、②是否忠實實作下方領域規格、③風險與盲點。
> 文末第 8 節列出「請你重點挑戰」的清單。歡迎直接反駁任何決策。

---

## 1. 專案背景與硬性約束

- **產品**：本地端執行的命理綜合分析平台，以**紫微斗數為主軸**，輔以生命靈數、八宅命卦、馬雅曆 Kin，並將新增八字。
- **技術**：Vite + Vanilla JS（ES modules），無 React/Vue。相依：`iztro`（紫微）、`lunar-javascript`（曆法/八字）、`chart.js`（作圖）。執行環境改用 Bun 加速。
- **硬性約束**：
  1. **純本地、無後端、無資料庫** —— 保障隱私、易於本地部署。
  2. 因此**刻意捨棄需要星曆表（Swiss Ephemeris）的印度占星（印占）與西洋占星（西占）**，只做純前端能精確計算的系統。
  3. **可重用性優先**：計算核心要能被多人重複使用（見第 4 節）。
- **關鍵取捨**：下方第 3 節的領域規格是完整方法論，涵蓋八字/印占/西占/人類圖等**多個系統**；但本專案基於上述約束**只實作其子集**（紫微／靈數／命卦／Kin／八字）。印占/西占/人類圖**保留架構接縫但目前不輸出**。審計時請以「子集是否忠實、接縫是否留對」來衡量，而非要求全系統。

## 3. 領域規格（本平台要忠實實作的方法論）

### 區塊 G：雷達與透明計分

- **各系統計分範例**：
  - 印占｜行星力量圖：入廟／旺 +2、友宮 +1、中性 0、敵宮 −1、落陷 −2；角宮（Kendra）再 +1；D9 同法計一次，並列 D1／D9。*(本專案無印占，僅示意計分風格。)*
  - 靈數｜1–9 數字頻次圖：出生年月日各位數出現次數長條。
- **呈現格式**：可作圖→雷達圖；不可作圖→表格＋文字長條（例：`水 ████████░░ 78%`）。
- **解讀規範**：雷達是「**佔比**」不是「優劣」；高佔比同時列出該屬性的**資產面與負債面**；**允許多軸同高**——這正是本區塊存在目的：讓「同時很理性又很感性」以數據直接可見，而不被匯流敘事剪掉。

### 區塊 H：動靜分層與時期演化（v2.0）

核心前提：人會隨時間持續變化——必須先分清哪些部件描述「不變的結構」、哪些描述「流動的狀態」，兩者不可混談。

1. **四層動靜分類**：
   - **L0 恆定結構層（一生不變）**：本命八字原局、本命盤宮位、人類圖類型/權威、命卦、Kin、生命靈數。
   - **L1 慢變層（10–20 年章節）**：八字大運、紫微大限、印占 Vimshottari 大運——同一 L0 結構被不同章節「點亮」不同部位。
   - **L2 年變層**：流年干支、紫微流年四化、印占中運（antardasha）、西占行運。
   - **L3 情境切換層（不隨時間、隨場合）**：西占上升(對外)/太陽(核心)/月亮(私下)、八字十神在不同對象前顯隱、紫微命宮vs身宮/三方四正、人類圖開放中心。
2. **狀態切換表**：以 L3 為基礎，輸出「情境→上場部件→表現樣貌」。至少涵蓋：初識場合／親密關係穩定期／衝突當下／壓力低谷／工作場域。每格標來源部件，並標「⚠️待驗證假說，供當事人核對」。
3. **時期演化雷達**：把區塊 G 雷達沿 L1 時間軸重算：
   - 八字：本命五行雷達 → 疊加各步大運干支（各計 1.0）逐十年重算 → 五行佔比序列 + 用神接近度曲線。
   - 紫微：各大限以大限四化與大限宮位重新加權十二宮 → 每十年宮強雷達。
   - 輸出：每十年一組「雷達＋兩行摘要」＋一段整體演化敘事。
4. **動靜誠實條款**：L0 可寫「你是」；L1／L2 只能寫「這段時期你會偏向」；L3 只能寫「在某情境下你會」。禁止把流年狀態寫成人格本質，也禁止把本質寫成不可變判決。

### v2 方法論提醒（審計基準）

- **雷達陷阱**：計分規則不揭露，雷達就只是裝飾——務必印出每一軸的計分方式，否則無法覆核、無法跨報告比較。
- **G 與 H 配套**：只有 G＝靜態快照；只有 H＝有時間軸但缺量化。兩塊要一起上。
- **L3 最易編故事**：組合自由度最高、最易寫成星座專欄軟文——每格必須標來源部件＋待驗證，鼓勵當事人核對回傳進校準迴圈。
- **v1/v2 關係**：v1 找「所有時期都成立的匯流結構」，v2 呈現「結構在不同軸向與時期的展開」。望遠鏡 vs 顯微鏡，非取代。**先 v1 定骨架，再 v2 上血肉**，順序反過來會迷失在細節。

## 4. 架構決策（已定案 2026-07-11）

1. **可重用核心函式庫**：`core / engines / analysis / visualization` 為框架無關 ES module 核心；`ui` 只是消費核心的其中一個 App。他人可只 `import` 計算核心自建介面。對外公開 API + semver 版本控管。
2. **系統含八字 BaZi**（共 5 系統）：八字本地可算（`lunar-javascript` 有八字），且規格大量使用。
3. **型別策略 JSDoc + `checkJs`**：零建置得到 IDE 型別檢查，不遷移 TypeScript。

### 目錄結構

```
src/
  index.js            ← 唯一公開 API（barrel + analyze + VERSION）
  core/               analyze.js / BaseEngine ✅ / EngineRegistry ✅ / models(BirthData ✅, SystemResult ✅)
  engines/            Ziwei ✅ Numerology ✅ MingGua ✅ Dreamspell ✅ / BaZi ⏳ / index.js ✅
  analysis/           LayerClassifier ✅ / ScoringRules ✅ / RadarBuilder ⏳ / StateSwitchTable ⏳
                      / EvolutionCalculator ⏳ / HonestyGuard ⏳
  visualization/      ChartTheme ✅ / TextFallback ✅ / RadarChart ⏳ / BarChart ⏳
  ui/                 App（非核心）：InputForm / 各 Panel
docs/  tests/
```

### 對外契約：`analyze()` → `Report`

```js
import { analyze, BirthData, VERSION } from 'fortunetelling';
const report = analyze(
  { year:1991, month:10, day:5, hour:14, gender:'female', name:'…' },
  { asOf:'2026-07-11' }   // 時間層(L1/L2/演化)評估基準日，預設今天
);
```

| `Report` 欄位 | 對應規格 | 內容 |
|---|---|---|
| `version/generatedAt/asOf/input` | — | 中繼資料與正規化輸入 |
| `engines` | — | 各引擎原始 `SystemResult[]` |
| `layers` | 區塊 H① | 四層分類（L0–L3 分組） |
| `radars` | 區塊 G | 每雷達的軸、**佔比**、每軸**計分規則**、資產/負債面 |
| `stateTable` | 區塊 H② | 情境→上場部件→表現，每格標來源部件 + ⚠️待驗證 |
| `evolution` | 區塊 H③ | 沿 L1 逐十年雷達序列 + 整體演化敘事 |
| `honesty` | 區塊 H④ | 語言規則 + 違規稽核結果 |

### 引擎外掛契約（多人重用的核心）

每個系統 = 一個 `BaseEngine` 子類，宣告 `id`/`name`，實作 `_compute(birth)` 回傳 `SystemResult`；其中每個 `component` 帶 `category`，由 `LayerClassifier` 映射到 L0–L3。新增系統不改動核心，只需：新增引擎檔 → 補分層規則 → 登錄計分公式 → 註冊 → 加測試。引擎**不得** import UI/visualization。

## 5. 里程碑（依「先骨架、再血肉」順序）

- **A. 可重用核心骨架** ★ 先做：`index.js` 公開 API、`analyze()` 編排器、`package.json exports`、`jsconfig.json(checkJs)`、協作者指南、首批 `bun test`。產出：他人可 `import { analyze }` 拿到含 L0/L1/L2 的報告（G/H 進階欄位先給穩定空殼）。
- **B. BaZiEngine**：四柱/日主/十神/五行分佈(L0)、大運(L1)、流年(L2)。`LayerClassifier` 已預埋 bazi 規則。
- **C. 區塊 G**：`RadarBuilder`（佔比，每軸獨立 0–100%，**允許多軸同高**，不強制加總 100）；每軸印計分規則；高佔比並列資產/負債；`RadarChart`/`BarChart` + 既有 `TextFallback` 降級。
- **D. 區塊 H**：`HonestyGuard`（依層級鎖語氣）、`StateSwitchTable`（5 情境，標來源+待驗證）、`EvolutionCalculator`（八字五行序列+用神曲線；紫微逐大限宮強；每十年雷達+摘要+整體敘事）。
- **E. UI 整合**：各 Panel 只吃 `Report`。

## 6. 已知風險與誠實但書

1. **Dreamspell Kin 錨定歧義**：實作錨定 1987-07-26 = Kin 34（可用單一公式同時重現 José Argüelles 的 Kin 11 與線上計算機當日值）；但業界另有把該日視為 Kin 1 的慣例。屬可設定的約定。
2. **命卦立春邊界簡化**：目前以 Feb-4 近似 立春 判斷跨年，未用精確節氣時刻（`BirthData.solarTermInfo` 可日後精修）。極少數生於立春當日者可能誤判。
3. **雷達佔比正規化方法未定案**：規格要求「佔比、可多軸同高」。目前傾向「每軸相對其自身理論上限做 0–100%」，而非跨軸 softmax（後者會強制競爭、違反多軸同高）。此為關鍵設計點，見第 8 節。
4. **跨系統雷達軸的可比性**：不同系統（紫微宮強 vs 八字五行 vs 靈數頻次）本質不同尺度，強行併入同一張雷達可能誤導。傾向**每系統各自一張雷達**，而非強行合併。待審計意見。
5. **紫微亮度權重**：採 廟7/旺6/得5/利4/平3/不2/陷1 線性；三方四正加權尚未實作（規劃於 RadarBuilder）。權重為可調參數，非命理權威定論。
6. **子集實作**：印占/西占/人類圖不輸出，故規格中依賴它們的 L3 情境（如西占上升/太陽/月亮）目前**缺對應部件**；狀態切換表初期會較稀疏，主要由紫微（命宮/身宮/三方四正）與八字十神支撐。

## 7. 非目標（明確不做）

- 不做後端、不做帳號、不做雲端儲存。
- 不引入星曆表；不做印占/西占的實際計算（僅留接縫）。
- 不宣稱命理為事實；`HonestyGuard` 與「待驗證」標註即為此防線。

## 8. 請審計者重點挑戰

1. **可重用性**：以「單一 `analyze()` + `Report` 契約 + 引擎外掛」作為對外 API，對多人重用是否足夠？公開面是否過大或過小？版本化策略是否需要更明確（例如 `Report.schemaVersion`）？
2. **雷達正規化（風險 3）**：每軸相對自身上限做 0–100% 是否正確體現「佔比非優劣、允許多軸同高」？有無更好的量化方式同時滿足「可跨報告比較」？
3. **跨系統雷達（風險 4）**：該「每系統一張雷達」還是「合併大雷達」？如何避免不同尺度誤導？
4. **規格忠實度**：第 5 節里程碑 C/D 是否完整覆蓋區塊 G/H？特別是「每軸印計分規則」「資產/負債並列」「L3 每格標來源+待驗證」「演化敘事」有無遺漏。
5. **誠實條款可執行性**：靠 `HonestyGuard` 在輸出層校驗語氣（你是／這段時期／在某情境下）是否可靠？是否該在資料層就把每個部件綁定其允許句式，而非事後校驗？
6. **子集造成的傾斜（風險 6）**：L3 情境表在缺西占/人類圖下會偏薄，是否應調整情境設計或明示「本平台版本」的侷限？
7. **測試策略**：以 `bun test` 對「已知命盤/命卦/Kin 向量」做黃金測試，對計分/分層做契約測試——對這類「無客觀真值」的命理輸出，什麼樣的測試才有意義？
8. **有無根本性的架構誤區**或更簡潔的做法？歡迎直接反對「可重用核心函式庫」這個方向本身。
